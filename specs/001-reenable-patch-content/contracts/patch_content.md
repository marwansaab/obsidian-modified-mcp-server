# Contract: `patch_content` MCP Tool

**Branch**: `001-reenable-patch-content` | **Date**: 2026-04-26

This document is the authoritative contract for what the MCP server
exposes to LLM clients via this tool. The runtime implementation in
`src/tools/patch-content/` MUST agree with everything below; tests in
`tests/tools/patch-content/` MUST verify each assertion.

---

## 1. Tool registration

When an MCP client calls `tools/list`, the response MUST include one
`Tool` entry with the following shape:

| Field | Value |
|---|---|
| `name` | `"patch_content"` |
| `description` | A human-readable string that includes **all three** of: (a) "heading targets require the full `H1::H2[::H3...]` path form", (b) "top-level headings (no parent) are unreachable through this tool", (c) "headings whose literal text contains `::` are unreachable through this tool". Exact prose is at the implementer's discretion. The **testable phrases** that the registration test (C12) asserts are present (case-insensitive substring match): (i) `H1::H2`, (ii) `top-level`, (iii) `literal text contains`. The third phrase is what distinguishes a description that documents the literal-`::` unreachable case from one that does not — a bare `::` substring would be trivially satisfied by `H1::H2` and would not actually verify constraint (c). |
| `inputSchema` | JSON Schema generated by `zod-to-json-schema` from the zod schema in `src/tools/patch-content/schema.ts`. See §2 below for the equivalent JSON Schema; the implementation MUST use the generator, not hand-written JSON. |

---

## 2. Input schema (equivalent JSON Schema)

This is the **shape the generator MUST produce** from the zod schema. It
is illustrative; the source of truth is the zod schema itself.

```json
{
  "type": "object",
  "properties": {
    "filepath": {
      "type": "string",
      "minLength": 1,
      "description": "Path to the file (relative to vault root)."
    },
    "operation": {
      "type": "string",
      "enum": ["append", "prepend", "replace"],
      "description": "How to insert the content relative to the target."
    },
    "targetType": {
      "type": "string",
      "enum": ["heading", "block", "frontmatter"],
      "description": "Type of target to locate."
    },
    "target": {
      "type": "string",
      "minLength": 1,
      "description": "When targetType is 'heading', the value MUST be a full path: at least two non-empty segments separated by '::'. Top-level headings and headings whose literal text contains '::' are unreachable through this tool."
    },
    "content": {
      "type": "string",
      "description": "The markdown content to insert."
    },
    "vaultId": {
      "type": "string",
      "description": "Optional vault ID (defaults to configured default vault)."
    }
  },
  "required": ["filepath", "operation", "targetType", "target", "content"]
}
```

**Note**: the heading-path rule is stated in the `target` field
description and enforced at runtime by the wrapper's
`isValidHeadingPath` predicate. It is intentionally not encoded as a
JSON Schema `pattern` (regex), because (a) the rule depends on
`targetType`, which JSON Schema cannot conditionally express cleanly,
and (b) the rule is structural across `::` segments and a regex would
obscure the intent.

---

## 3. Successful response

When the patch succeeds (upstream returns 2xx), the tool MUST return:

```json
{
  "content": [
    { "type": "text", "text": "Content patched successfully" }
  ]
}
```

The `isError` field is absent (or explicitly `false`). The text payload
matches the convention used by sibling write tools (`append_content`,
`put_content`).

---

## 4. Validation-failure response (wrapper-side rejection)

When the wrapper rejects an input before issuing any HTTP call, the
tool MUST return:

```json
{
  "content": [
    { "type": "text", "text": "Error: <message>" }
  ],
  "isError": true
}
```

The `<message>` substring MUST contain the three required components
specified in [data-model.md §`WrapperValidationError`](../data-model.md):

1. The rule name (verbatim string).
2. `received: "<offending value>"`.
3. `e.g., "<corrected example>"`.

A test in `tests/tools/patch-content/schema.test.ts` MUST assert all
three substrings are present in the message of every rejection path.

---

## 5. Upstream-failure response (post-validation, network or HTTP error)

When the upstream Local REST API returns non-2xx, or the network call
itself fails, the tool MUST return:

```json
{
  "content": [
    { "type": "text", "text": "Error: Obsidian API Error <code>: <upstream-message>" }
  ],
  "isError": true
}
```

- `<code>` is the upstream `errorCode` (from response body if present),
  else the HTTP status code (e.g., `404`, `401`), else `-1` for
  transport-level failures (no response received).
- `<upstream-message>` is the upstream `message` (from response body if
  present) or the underlying axios error message.

This format is produced by the existing `safeCall` helper in
`src/services/obsidian-rest.ts:35-47` plus the existing top-level
handler in `src/index.ts:250-257`. The new wrapper MUST NOT catch and
suppress these errors.

---

## 6. Forbidden behaviors (explicit non-contracts)

The wrapper MUST NOT:

- Return a successful response when the upstream failed (no silent
  fallbacks; Constitution Principle IV).
- Trim, normalize, or otherwise transform the `target` string before
  forwarding to upstream.
- Apply heading-path validation when `targetType` is `block` or
  `frontmatter`.
- Make any HTTP call when wrapper-side validation fails.
- Attempt retries on transient failures (PATCH is non-idempotent;
  see research R6).

---

## 7. Test matrix (contract-level)

Each row maps to one test in
`tests/tools/patch-content/{schema|handler}.test.ts`.

| ID | Scenario | Expected outcome |
|---|---|---|
| C1 | `targetType=heading`, valid 2-segment path, upstream returns 2xx | Successful response per §3 |
| C2 | `targetType=heading`, `target="Action Items"` (bare) | Validation error per §4; **no HTTP call made** (verified at the schema level) |
| C2b | C2 driven through the handler (not the schema) with `nock.disableNetConnect()` | Validation error per §4; the handler must call `assertValidPatchRequest` before `rest.patchContent` so no `NetConnectNotAllowedError` is observed (verifies SC-001's "zero requests reaching the upstream" claim at the integration boundary, not just at the validator) |
| C3 | `targetType=heading`, `target="A::B::"` (trailing empty) | Validation error per §4; no HTTP call |
| C4 | `targetType=heading`, `target="::A::B"` (leading empty) | Validation error per §4; no HTTP call |
| C5 | `targetType=heading`, `target="A::::B"` (middle empty) | Validation error per §4; no HTTP call |
| C6 | `targetType=heading`, `target=""` (empty) | Validation error per §4; no HTTP call |
| C7 | `targetType=heading`, `target="   "` (whitespace-only) | Validation error per §4; no HTTP call |
| C8 | `targetType=block`, any `target` | No heading-path validation; HTTP call made; behavior pass-through |
| C9 | `targetType=frontmatter`, any `target` | No heading-path validation; HTTP call made; behavior pass-through |
| C10 | Valid input, upstream returns 404 | Upstream-failure response per §5; preserves status code and message |
| C10b | Valid input, upstream returns 401 (auth failure) | Upstream-failure response per §5; preserves status code and message — explicitly exercises spec User Story 3's "authentication failure" scenario |
| C11 | Valid input, upstream unreachable (`ECONNREFUSED`) | Upstream-failure response per §5 with code `-1` and underlying message |
| C12 | `tools/list` is called | Response contains an entry with `name: "patch_content"` and a `description` whose case-insensitive content contains all three testable phrases listed in §1: `H1::H2`, `top-level`, `literal text contains` |

C1, C8, and C9 all assert the upstream HTTP **headers**
(`Operation`, `Target-Type`, `Target` URL-encoded,
`Content-Type: text/markdown`) via the test runner's header-matching
helpers. Without those assertions, the tests could silently pass even
if `rest.patchContent` stopped sending one of the documented headers
— undermining FR-005. Asserting on all three target-type variants
(heading, block, frontmatter) is defensive against a hypothetical
future `rest.patchContent` regression that introduces conditional
header logic per target type.

C1–C7 satisfy FR-009 plus the additional whitespace and middle-empty
cases identified in spec Edge Cases. C2b verifies that the handler
itself enforces the validate-before-HTTP order (FR-003 / SC-001 at
integration). C8–C9 verify FR-006 (block / frontmatter pass-through).
C10, C10b, and C11 verify FR-007 (Constitution IV). C12 verifies
FR-001.
