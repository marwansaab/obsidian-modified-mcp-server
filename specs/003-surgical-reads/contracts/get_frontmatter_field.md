# Contract: `get_frontmatter_field` MCP Tool

**Branch**: `003-surgical-reads` | **Date**: 2026-04-26

This document is the authoritative contract for what the MCP server
exposes to LLM clients via this tool. The runtime implementation in
`src/tools/surgical-reads/` MUST agree with everything below; tests in
`tests/tools/surgical-reads/` MUST verify each assertion.

---

## 1. Tool registration

When an MCP client calls `tools/list`, the response MUST include one
`Tool` entry with the following shape:

| Field | Value |
|---|---|
| `name` | `"get_frontmatter_field"` |
| `description` | A human-readable string that includes **all three** of: (a) "Returns the named frontmatter field's value with its original type preserved — string, number, boolean, array, object, or null", (b) "If the field or the note does not exist, the upstream's 4xx error is propagated unchanged", (c) "To read all frontmatter fields at once, use `get_file_contents`". Exact prose is at the implementer's discretion. The **testable phrases** that the registration test asserts are present (case-insensitive substring match): (i) `original type preserved`, (ii) `4xx`, (iii) `get_file_contents`. Phrase (i) verifies the typed-value contract from clarification Q2 is named explicitly; phrase (ii) verifies the missing-field-as-error contract; phrase (iii) verifies the all-frontmatter fallback is named. |
| `inputSchema` | JSON Schema generated by `zod-to-json-schema` from the zod schema in `src/tools/surgical-reads/schema.ts`. See §2 below for the equivalent JSON Schema; the implementation MUST use the generator, not hand-written JSON. |

---

## 2. Input schema (equivalent JSON Schema)

This is the **shape the generator MUST produce** from the zod schema. It
is illustrative; the source of truth is the zod schema itself.

```json
{
  "type": "object",
  "properties": {
    "filepath": {
      "type": "string",
      "minLength": 1,
      "description": "Path to the file (relative to vault root)."
    },
    "field": {
      "type": "string",
      "minLength": 1,
      "description": "The name of the single frontmatter field to read. Must be non-empty after trimming whitespace."
    },
    "vaultId": {
      "type": "string",
      "description": "Optional vault ID (defaults to configured default vault)."
    }
  },
  "required": ["filepath", "field"]
}
```

The "non-empty after trim" constraint is enforced at runtime by a zod
`.refine()` predicate. JSON Schema's standard vocabulary cannot express
"non-whitespace-only," so the constraint is documented in the field
description and verified by tests, not by schema-driven validation
alone.

---

## 3. Successful response

When the upstream returns 2xx, the tool MUST return:

```json
{
  "content": [
    {
      "type": "text",
      "text": "{\"value\":<json-encoding-of-decoded-value>}"
    }
  ]
}
```

The `isError` field is absent (or explicitly `false`). The wrapper:

1. Decodes the upstream response body via `JSON.parse(response.data)`
   (or relies on axios's automatic JSON decoding; either is acceptable
   provided the resulting JS value preserves the original type).
2. Wraps the decoded value as `{ value: <decoded> }`.
3. Calls `JSON.stringify` on that wrapper object and places it in
   `content[0].text`.

The result is that callers parse `content[0].text` once as JSON to
recover a typed `value`. Concrete examples:

| Frontmatter source (YAML, conceptually) | Upstream JSON body | Wrapper output `text` |
|---|---|---|
| `status: in-progress` | `"in-progress"` | `{"value":"in-progress"}` |
| `count: 5` | `5` | `{"value":5}` |
| `published: true` | `true` | `{"value":true}` |
| `aliases: [a, b]` | `["a","b"]` | `{"value":["a","b"]}` |
| `meta: {x: 1}` | `{"x":1}` | `{"value":{"x":1}}` |
| `archived: null` | `null` | `{"value":null}` |

The `value: null` row is a present-but-null field, distinct from a
missing field (which is §5 below).

## 4. Validation-failure response (wrapper-side rejection)

When the wrapper rejects the input before issuing any HTTP call, the
tool MUST return:

```json
{
  "content": [
    { "type": "text", "text": "Error: <message>" }
  ],
  "isError": true
}
```

For this tool the only wrapper-side rejections are:

- `field` missing or not a string → zod error naming `field`.
- `field` is an empty string → zod error naming `field` (caught by
  `.min(1)`).
- `field` is whitespace-only after `.trim()` (e.g., `"   "`) → zod
  error from the custom `.refine()`; the message contains the phrase
  `field must not be whitespace-only` (or equivalent zod refinement
  text) and names the `field` field path.
- `filepath` missing, not a string, or empty → zod error naming
  `filepath`.

A test in `tests/tools/surgical-reads/schema.test.ts` MUST assert that
each rejection's message contains the relevant field name. No
path-rule three-substring check applies to this tool.

---

## 5. Upstream-failure response (post-validation, network or HTTP error)

When the upstream Local REST API returns non-2xx, or the network call
itself fails, the tool MUST return:

```json
{
  "content": [
    { "type": "text", "text": "Error: Obsidian API Error <code>: <upstream-message>" }
  ],
  "isError": true
}
```

- `<code>` is the upstream `errorCode` (from response body if present),
  else the HTTP status code (e.g., `404`, `401`), else `-1` for
  transport-level failures (no response received).
- `<upstream-message>` is the upstream `message` (from response body
  if present) or the underlying axios error message.

This format is produced by the existing `safeCall` helper in
[src/services/obsidian-rest.ts:35-47](../../src/services/obsidian-rest.ts#L35-L47)
plus the existing top-level handler in
[src/index.ts:251-258](../../src/index.ts#L251-L258). The new wrapper
MUST NOT catch and suppress these errors.

A field-not-found case (upstream 404) and a present-but-null field
case (upstream 200 with body `null`) are distinguishable to the
caller: the former is `isError: true` with an upstream message; the
latter is success with `text: '{"value":null}'`. This distinction is
load-bearing per spec FR-009 + clarification Q2.

---

## 6. Forbidden behaviors (explicit non-contracts)

The wrapper MUST NOT:

- Return a successful response when the upstream failed (no silent
  fallbacks; Constitution Principle IV).
- Stringify, coerce, or otherwise transform the decoded value before
  wrapping it. A frontmatter `count: 5` MUST be `{"value":5}`, not
  `{"value":"5"}`.
- Collapse `null` into "missing" or empty string. A present-but-null
  value MUST be `{"value":null}`.
- Read or parse the underlying note's content client-side. The only
  decoding allowed is `JSON.parse` of the upstream response envelope.
- Make any HTTP call when wrapper-side validation fails.
- Attempt retries on transient failures.

## 7. Test matrix (contract-level)

Each row maps to one test in
`tests/tools/surgical-reads/{schema|frontmatter-handler|registration}.test.ts`.

| ID | Scenario | Expected outcome |
|---|---|---|
| F1 | `field="status"`, upstream returns 200 with body `"in-progress"` (JSON-encoded string) | Successful response per §3; `text === '{"value":"in-progress"}'`. nock asserts the URL path is `/vault/note.md/frontmatter/status`. |
| F2 | `field="count"`, upstream returns 200 with body `5` (JSON number) | Successful response per §3; `text === '{"value":5}'` (not `'{"value":"5"}'`) |
| F3 | `field="published"`, upstream returns 200 with body `true` (JSON boolean) | Successful response per §3; `text === '{"value":true}'` |
| F4 | `field="aliases"`, upstream returns 200 with body `["a","b"]` (JSON array) | Successful response per §3; `text === '{"value":["a","b"]}'` |
| F5 | `field="meta"`, upstream returns 200 with body `{"x":1}` (JSON object) | Successful response per §3; `text === '{"value":{"x":1}}'` |
| F6 | `field="archived"`, upstream returns 200 with body `null` (JSON null) | Successful response per §3; `text === '{"value":null}'` (verifies Q2 clarification — null is present-and-typed, not missing) |
| F7 | `field=""` (empty) | Validation error per §4 caught by zod `.min(1)`; message names `field`; no HTTP call |
| F8 | `field="   "` (whitespace-only) | Validation error per §4 caught by zod `.refine()`; message names `field` and includes refinement text; no HTTP call |
| F9 | Special characters in `field` (e.g., `field="my:custom"`) and a normal value | Verifies URL-encoding: nock asserts the URL path is `/vault/note.md/frontmatter/my%3Acustom`. Verifies R8. |
| F9b | Special characters in **filepath** (`filepath="Folder With Spaces/note name.md"`, `field="status"`) | Verifies filepath URL-encoding: nock asserts the URL path is `/vault/Folder%20With%20Spaces/note%20name.md/frontmatter/status` — same per-component encoding contract as H1d (the `/` between folder and filename is preserved, while spaces are percent-encoded to `%20`). |
| F10 | Valid input, upstream returns 404 with `{errorCode: 40400, message: "Frontmatter field not found"}` | Upstream-failure response per §5; preserves `40400` and message. Distinct from F6. |
| F11 | Valid input, upstream returns 401 with `{errorCode: 40100, message: "Invalid API key"}` | Upstream-failure response per §5; preserves status code and message |
| F12 | Valid input, upstream unreachable (`ECONNREFUSED` via `nock.disableNetConnect()`) | Upstream-failure response per §5 with code `-1` |
| FR | `tools/list` is called | Response contains an entry with `name: "get_frontmatter_field"` and a `description` whose case-insensitive content contains all three testable phrases listed in §1: `original type preserved`, `4xx`, `get_file_contents` |

F1–F6 jointly verify FR-008's typed-value contract and clarification
Q2. F2–F6 are the load-bearing rows: a buggy implementation that
stringified everything would still pass F1 (the source value is
already a string) but would fail F2–F6. F6 specifically verifies the
`null`-as-typed-value contract.

F7 and F8 satisfy FR-007. F9 satisfies FR-011(b)'s URL-encoding
requirement. F10/F11/F12 verify FR-009 (Constitution IV). F10's
distinctness from F6 is the test that catches a regression where the
wrapper might be tempted to collapse `null` into "missing" — see
spec FR-009 + clarification Q2.

FR verifies FR-006 at the integration boundary.
