# Contract: `get_heading_contents` MCP Tool

**Branch**: `003-surgical-reads` | **Date**: 2026-04-26

This document is the authoritative contract for what the MCP server
exposes to LLM clients via this tool. The runtime implementation in
`src/tools/surgical-reads/` MUST agree with everything below; tests in
`tests/tools/surgical-reads/` MUST verify each assertion.

---

## 1. Tool registration

When an MCP client calls `tools/list`, the response MUST include one
`Tool` entry with the following shape:

| Field | Value |
|---|---|
| `name` | `"get_heading_contents"` |
| `description` | A human-readable string that includes **all five** of: (a) "heading targets require the full `H1::H2[::H3...]` path form", (b) "top-level headings (no parent) are unreachable through this tool", (c) "headings whose literal text contains `::` are unreachable through this tool", (d) for both unreachable cases, the documented fallback is `get_file_contents` followed by client-side slicing, and (e) "Returns the raw markdown body content under the targeted heading. Frontmatter, tags, and file metadata are not included — use `get_file_contents` for the whole note or `get_frontmatter_field` for individual frontmatter values." Exact prose is at the implementer's discretion. The **testable phrases** that the registration test asserts are present (case-insensitive substring match): (i) `h1::h2`, (ii) `top-level`, (iii) `literal text contains`, (iv) `get_file_contents`, (v) `frontmatter, tags`. Phrase (iv) verifies the fallback is named; phrase (v) verifies the metadata-exclusion clause from clarification Q1 is present (a generic "raw markdown body" mention without naming the excluded categories would not satisfy this). |
| `inputSchema` | JSON Schema generated by `zod-to-json-schema` from the zod schema in `src/tools/surgical-reads/schema.ts`. See §2 below for the equivalent JSON Schema; the implementation MUST use the generator, not hand-written JSON. |

---

## 2. Input schema (equivalent JSON Schema)

This is the **shape the generator MUST produce** from the zod schema. It
is illustrative; the source of truth is the zod schema itself.

```json
{
  "type": "object",
  "properties": {
    "filepath": {
      "type": "string",
      "minLength": 1,
      "description": "Path to the file (relative to vault root)."
    },
    "heading": {
      "type": "string",
      "minLength": 1,
      "description": "Full heading path: at least two non-empty segments separated by '::' (i.e., the H1::H2[::H3...] form). Top-level headings and headings whose literal text contains '::' are unreachable through this tool — fall back to get_file_contents for those cases."
    },
    "vaultId": {
      "type": "string",
      "description": "Optional vault ID (defaults to configured default vault)."
    }
  },
  "required": ["filepath", "heading"]
}
```

**Note**: the heading-path rule is stated in the `heading` field
description and enforced at runtime by the wrapper's
`isValidHeadingPath` predicate (imported from
`src/tools/patch-content/schema.ts`). It is intentionally not encoded
as a JSON Schema `pattern` (regex), for the same reasons as
`patch_content`: the rule is structural across `::` segments and a
regex would obscure intent.

---

## 3. Successful response

When the upstream returns 2xx, the tool MUST return:

```json
{
  "content": [
    { "type": "text", "text": "<raw markdown body of the targeted heading section>" }
  ]
}
```

The `isError` field is absent (or explicitly `false`). The response
body bytes returned by the upstream — whatever they are, including
empty string `""` — are placed verbatim in `content[0].text`. The
wrapper does not slice, trim, re-wrap, or otherwise modify them.

For an upstream 2xx response with empty body, the response is:

```json
{ "content": [{ "type": "text", "text": "" }] }
```

This is well-formed and explicitly distinct from a `404`/error
response. See spec Edge Cases.

---

## 4. Validation-failure response (wrapper-side rejection)

When the wrapper rejects the input before issuing any HTTP call, the
tool MUST return:

```json
{
  "content": [
    { "type": "text", "text": "Error: <message>" }
  ],
  "isError": true
}
```

For heading-rule rejections, the `<message>` substring MUST contain
the three required components specified in [data-model.md §`WrapperValidationError`](../data-model.md):

1. The rule name (verbatim string): `heading targets must use the full H1::H2[::H3...] path`
2. `received: "<offending value>"`.
3. `e.g., "<corrected example>"`.

These are the same three substrings the `patch_content` rejection test
asserts (FR-004's "must match" requirement). A test in
`tests/tools/surgical-reads/schema.test.ts` MUST assert all three
substrings are present in the message of every heading-rule rejection
path.

For zod-level rejections (e.g., missing `filepath`, wrong type), the
message contains zod's standard field-path output — no separate
assertion is required beyond "the field name appears in the message."

---

## 5. Upstream-failure response (post-validation, network or HTTP error)

When the upstream Local REST API returns non-2xx, or the network call
itself fails, the tool MUST return:

```json
{
  "content": [
    { "type": "text", "text": "Error: Obsidian API Error <code>: <upstream-message>" }
  ],
  "isError": true
}
```

- `<code>` is the upstream `errorCode` (from response body if present),
  else the HTTP status code (e.g., `404`, `401`), else `-1` for
  transport-level failures (no response received).
- `<upstream-message>` is the upstream `message` (from response body
  if present) or the underlying axios error message.

This format is produced by the existing `safeCall` helper in
[src/services/obsidian-rest.ts:35-47](../../src/services/obsidian-rest.ts#L35-L47)
plus the existing top-level handler in
[src/index.ts:251-258](../../src/index.ts#L251-L258). The new wrapper
MUST NOT catch and suppress these errors.

---

## 6. Forbidden behaviors (explicit non-contracts)

The wrapper MUST NOT:

- Return a successful response when the upstream failed (no silent
  fallbacks; Constitution Principle IV).
- Trim, normalize, or otherwise transform the `heading` string before
  validating or before forwarding to upstream.
- Synthesize a "not found" error from a 2xx empty body — empty body
  is a valid heading-section content for an empty section.
- Request `application/vnd.olrapi.note+json` or any other Accept type
  that returns the JSON envelope. Only `text/markdown` is requested.
- Include frontmatter, tags, or file metadata in the response.
- Make any HTTP call when wrapper-side validation fails.
- Define its own copy of `isValidHeadingPath`. The predicate MUST
  be imported from `src/tools/patch-content/schema.ts` (FR-003 single
  source of truth).
- Attempt retries on transient failures.

## 7. Test matrix (contract-level)

Each row maps to one test in
`tests/tools/surgical-reads/{schema|heading-handler|registration}.test.ts`.

| ID | Scenario | Expected outcome |
|---|---|---|
| H1 | Valid 2-segment heading path, upstream returns 2xx with body `"- item one\n- item two\n"` | Successful response per §3 with `text` equal to that body verbatim |
| H1b | Valid 2-segment path, upstream returns 2xx with empty body | Successful response per §3 with `text` equal to `""` (no synthesized error) |
| H1c | Valid 2-segment path with special characters in a segment (e.g., `Project::Q3 / Plan`), upstream returns 2xx | Verifies URL-encoding: nock asserts the URL path is `/vault/note.md/heading/Project/Q3%20%2F%20Plan` — segment split on `::`, then each segment `encodeURIComponent`-ed, then re-joined with `/`. Verifies R8. |
| H1d | Valid 2-segment heading path with special characters in the **filepath** (`filepath="Folder With Spaces/note name.md"`, `heading="A::B"`), upstream returns 2xx | Verifies filepath URL-encoding: nock asserts the URL path is `/vault/Folder%20With%20Spaces/note%20name.md/heading/A/B` — the `/` between folder and filename is preserved as a literal path-segment separator, while spaces in each component are percent-encoded. Verifies the per-component encoding from R8 on the filepath side (H1c verifies it on the heading-segment side). |
| H2 | `heading="Action Items"` (bare) | Validation error per §4 with the three required substrings; **no HTTP call made** (verified at schema level via `assertValidHeadingPathTarget`) |
| H2b | H2 driven through the handler with `nock.disableNetConnect()` | Validation error per §4; the handler must call the validator before `rest.getHeadingContents` so no `NetConnectNotAllowedError` is observed (verifies SC-001 at the integration boundary) |
| H3 | `heading="A::B::"` (trailing empty) | Validation error per §4; no HTTP call |
| H4 | `heading="::A::B"` (leading empty) | Validation error per §4; no HTTP call |
| H5 | `heading="A::::B"` (middle empty) | Validation error per §4; no HTTP call |
| H6 | `heading=""` (empty) | Validation error caught by zod `.min(1)`; message names `heading` field; no HTTP call |
| H7 | `heading="   "` (whitespace-only) | Validation error per §4 (passes `min(1)`, fails `isValidHeadingPath`); no HTTP call |
| H8 | Valid input, upstream returns 404 with `{errorCode: 40400, message: "File or heading not found"}` | Upstream-failure response per §5; preserves `40400` and `"File or heading not found"` |
| H9 | Valid input, upstream returns 401 with `{errorCode: 40100, message: "Invalid API key"}` | Upstream-failure response per §5; preserves status code and message |
| H10 | Valid input, upstream unreachable (`ECONNREFUSED` via `nock.disableNetConnect()`) | Upstream-failure response per §5 with code `-1` |
| HR | `tools/list` is called | Response contains an entry with `name: "get_heading_contents"` and a `description` whose case-insensitive content contains all five testable phrases listed in §1: `h1::h2`, `top-level`, `literal text contains`, `get_file_contents`, `frontmatter, tags` |

H1, H1b, H1c, H1d assert the upstream HTTP **path and Accept header** via
nock's path-matching and `.matchHeader` helpers. Without these
assertions, the tests could silently pass even if `rest.getHeadingContents`
stopped sending `Accept: text/markdown` or constructed the URL
incorrectly — undermining FR-005.

H1c and H1d satisfy the URL-encoding case in spec FR-011(a) — H1c on
the heading-segment side, H1d on the filepath side. H2 through H7
satisfy FR-003 + FR-004. H8/H9/H10 verify FR-009 (Constitution IV).
HR verifies FR-001 at the integration boundary.
