# Feature Specification: Safe Rename Tool

**Feature Branch**: `012-safe-rename`  
**Created**: 2026-05-02  
**Status**: Draft (Option B — post-spike redesign; implementation gated on backlog item 25 / `find_and_replace`)  
**Input**: User description: "Safe Rename Tool — A new MCP tool rename_file that renames a file in the vault while preserving wikilink integrity vault-wide. The tool accepts an old_path and a new_path. It dispatches Obsidian's built-in 'Rename file' command via the existing execute_command infrastructure (POST /commands/{commandId}/) rather than issuing a filesystem-level rename — this triggers Obsidian's 'Automatically update internal links' behaviour so every [[wikilink]] referencing the old name is rewritten in the same operation."

> **Original Option-A design (Obsidian command dispatch) was empirically shown to be infeasible by the 2026-05-02 T002 feasibility spike.** Stock Obsidian's "Rename file" / "file-explorer:move-file" commands open UI inputs and cannot perform a programmatic rename when dispatched headlessly via `POST /commands/{commandId}/`. See [research.md §R5](./research.md) for the spike result and [§"Clarifications"](#clarifications) for the supersession log. The tool's user-facing surface (name, parameters, vault-wide-link-integrity guarantee) is preserved; the internal mechanism is **Option B** — composition over filesystem primitives + the vault-wide `find_and_replace` tool from Tier 2 backlog item 25.

## Clarifications

### Session 2026-05-02 (initial)

- Q: Error contract when source is missing or destination collides — pre-flight detection, pure delegation, or hybrid? → A: Pure delegation. The tool dispatches the Obsidian command unconditionally and propagates the underlying response verbatim on failure; error messages and conflict-naming detail are whatever Obsidian returns, not constructed by the tool itself.
- Q: Which vault paths are in scope — markdown notes only, notes + attachments, or also folders? → A: Notes + attachments (any path Obsidian treats as a file); folders are explicitly out of scope and must be rejected. Attachment handling is inherited at zero cost from Obsidian's "Rename file" command (which rewrites `![[image.png]]` embed links the same way it rewrites `[[wikilinks]]`). Folder renames have different semantics — separate command id, prefix-based link rewriting, distinct failure modes — and belong in a future, separately-named `rename_folder` tool, not folded under this name.
- Q: Cross-folder rename when the destination folder doesn't exist — fail and surface, auto-create unconditionally, or opt-in flag? → A: Strict — fail and surface the underlying Obsidian error; do not auto-create parent folders and do not add a `create_parents`-style flag. Folder creation is a separate, named capability that callers compose explicitly; folding it into `rename_file` would expand the contract beyond what the tool name advertises and add a code path the tool does not otherwise need.

### Session 2026-05-02 (post-spike, Option-B redesign — supersedes the relevant clauses above)

- **T002 spike outcome → Option B chosen.** The 2026-05-02 spike against the deployed `@marwansaab/obsidian-modified-mcp-server@0.5.0` connected to a live Obsidian instance found that neither `workspace:edit-file-title` nor `file-explorer:move-file`, when dispatched via `POST /commands/{commandId}/`, performs a programmatic rename. Both commands open Obsidian UI inputs and silently no-op under headless dispatch. No body shape changes the outcome — the commands are hardcoded for in-app interactive use. Per [research.md §R5](./research.md) the failure-action is to escalate; the resolution chosen by the user is **Option B**: replace the Obsidian-command dispatch with a wrapper-side composition over filesystem primitives plus the vault-wide `find_and_replace` tool (Tier 2 backlog item 25). Item 4's user-facing surface is preserved; the implementation pathway is rebuilt. Restoring Option A (Obsidian-managed rename) is captured separately as backlog item 28 — deferred pending an upstream `coddingtonbear/obsidian-local-rest-api` plugin enhancement that exposes a programmatic file-rename endpoint with body parameters; out of this project's direct control.
- **Q1 partially superseded — FR-006 only.** Under Option B, the destination-collision check (FR-006) is now wrapper-side rather than upstream-delegated. The wrapper performs a `rest.getFileContents` pre-flight against `new_path` and constructs the collision error itself: `"destination already exists: <new_path>"`. This deviates from Q1's pure-delegation principle but is justified because `rest.putContent`'s default semantic is overwrite, which would silently violate FR-006's "MUST refuse … returning an error that names the conflict" contract. **Q1 STILL APPLIES to FR-007** (source-missing — `rest.getFileContents`'s upstream 404 propagates verbatim) **and to FR-012** (missing-parent-folder — `rest.listFilesInDir`'s upstream 404 propagates verbatim).
- **Q2 unchanged** — file scope (notes + attachments, no folders) holds. The folder-rejection path now manifests via `rest.getFileContents` returning a non-2xx for a folder path rather than via `openFile` failing.
- **Q3 unchanged** — no auto-create of missing parent folders, no `create_parents` flag. The check is now an active wrapper-side pre-flight (`rest.listFilesInDir(dirname(new_path))`) rather than upstream-delegated, but the rule and rationale are identical.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Rename a file and keep every wikilink intact (Priority: P1)

An LLM agent or human caller wants to rename a note in the vault — for example, moving `Inbox/draft.md` to `Projects/Project-X/overview.md` — and have every existing `[[draft]]` or `[[Inbox/draft]]` reference across the entire vault automatically updated to point at the new location. The caller invokes `rename_file` with the old and new paths and trusts that no incoming links will break.

**Why this priority**: This is the entire purpose of the feature. Vault-wide link integrity is the single guarantee that distinguishes this tool from a plain filesystem move. Without this story working, the tool has no reason to exist.

**Independent Test**: In a vault committed to clean git state, with a note `notes/alpha.md` and a separate note `index.md` containing `See [[alpha]] for details.`, call `rename_file({old_path: "notes/alpha.md", new_path: "notes/beta.md"})`. Verify (a) `notes/beta.md` exists with the original content of `notes/alpha.md`, (b) `notes/alpha.md` no longer exists, (c) `notes/index.md` now reads `See [[beta]] for details.`, (d) the response identifies both paths and includes a non-zero `wikilinkRewriteCounts.passA`.

**Acceptance Scenarios**:

1. **Given** a vault with `notes/alpha.md` and a separate note `index.md` containing the body `See [[alpha]] for details.`, **When** the caller invokes `rename_file` with `old_path="notes/alpha.md"` and `new_path="notes/beta.md"`, **Then** `notes/beta.md` exists, `notes/alpha.md` does not, and `index.md` reads `See [[beta]] for details.`
2. **Given** a vault where many notes link to `Inbox/draft.md` using a mix of `[[draft]]`, `[[draft|alias]]`, `[[draft#heading]]`, `[[draft#heading|alias]]`, `[[draft#^block-id]]`, `![[draft]]`, and `![[draft|alias]]` forms, **When** the caller renames the file to `Projects/Project-X/overview.md`, **Then** every variant in every referencing note is updated to the equivalent reference for the new basename, and user-supplied aliases are preserved.
3. **Given** a successful rename, **When** the tool returns, **Then** the response includes both paths plus the `wikilinkPassesRun` array (which passes executed) and `wikilinkRewriteCounts` per pass, so the caller can confirm what changed without a follow-up read.

---

### User Story 2 - Refuse to rename when the target already exists (Priority: P2)

A caller mistakenly issues a rename whose `new_path` collides with an existing file. The tool must not silently overwrite or merge — it must surface the conflict so the caller can choose a different name or explicitly delete the conflicting file first.

**Why this priority**: Silent overwrite would destroy user data. This safety property is required for the tool to be usable autonomously by an agent, but it is secondary to the core rename behaviour in Story 1.

**Independent Test**: In a vault containing both `a.md` and `b.md`, invoke `rename_file({old_path: "a.md", new_path: "b.md"})`. Verify that the tool returns a wrapper-constructed error of the form `"destination already exists: b.md"` and that both files remain untouched on disk.

**Acceptance Scenarios**:

1. **Given** `a.md` and `b.md` both exist in the vault, **When** the caller invokes `rename_file(old_path="a.md", new_path="b.md")`, **Then** the tool returns an error indicating the destination already exists (wrapper-constructed: `"destination already exists: b.md"`) and no files are modified.
2. **Given** the rename was rejected at the pre-flight stage (collision, missing source, missing parent folder, malformed path), **When** the caller inspects the vault afterwards, **Then** every file is in the same state as before the call — no partial rename, no partial link rewrite. (See SC-003 — atomicity holds for pre-flight rejections; mid-flight failures are explicitly best-effort.)

---

### User Story 3 - Make the precondition + integrity contract discoverable (Priority: P2)

An agent reading the MCP tool catalogue should be able to learn, from the tool description alone, the operational contract: that the rename is multi-step and not atomic, that the vault should be on clean git state before invocation as the rollback baseline, which wikilink shapes are reliably rewritten, and that the Obsidian "Automatically update internal links" setting is irrelevant under this implementation. The agent (or its operator) can then adopt the tool safely instead of discovering the gaps by hitting them in production.

**Why this priority**: The operational contract is non-trivial. If the description hides the non-atomicity, callers will assume failure leaves the vault unchanged — and lose data when a mid-flight failure leaves a partial state with no automated recovery. Surfacing the contract is what makes the trade-off acceptable.

**Independent Test**: Inspect the MCP tool list (e.g., via `tools/list`). Verify the `rename_file` description text contains all of: (a) the multi-step / non-atomic disclosure, (b) the git-clean precondition, (c) the reliably-rewritten wikilink shape list, (d) the explicit statement that the Obsidian "Automatically update internal links" setting is irrelevant.

**Acceptance Scenarios**:

1. **Given** any MCP client connected to this server, **When** it lists available tools, **Then** the description for `rename_file` discloses non-atomicity, the git-clean precondition, the wikilink shape coverage, and the irrelevance of the Obsidian setting.
2. **Given** the description is read by a human or an LLM, **When** they reason about whether the tool is safe to use, **Then** the description gives them enough information to know (i) to commit before invoking, (ii) which wikilink shapes will and won't be reliably updated, (iii) that the Obsidian setting need not be configured — without needing to read the source code or external docs.

### Edge Cases

- **Source file does not exist**: `old_path` points to a path the vault does not contain. The pre-flight `rest.getFileContents(old_path)` returns 404 and the wrapper propagates the upstream error verbatim (per FR-007). No file is created at `new_path`. Pre-flight rejection — atomicity holds.
- **Source and destination are identical**: `old_path == new_path`. The wrapper returns the FR-009 idempotent no-op result without issuing any REST call.
- **Destination folder does not exist**: `dirname(new_path)` is not an existing folder. The pre-flight `rest.listFilesInDir(dirname(new_path))` returns 404 and the wrapper propagates that error verbatim (per FR-012). No auto-create. Pre-flight rejection — atomicity holds.
- **Destination already exists**: `new_path` resolves to an existing file. The pre-flight `rest.getFileContents(new_path)` returns 200 and the wrapper constructs the FR-006 collision error (`"destination already exists: <new_path>"`). No file is read, written, rewritten, or deleted. Pre-flight rejection — atomicity holds.
- **Path outside the vault**: Either path resolves outside the focused vault. Each composed REST primitive (`getFileContents`, `putContent`, `listFilesInDir`, `findAndReplace`, `deleteFile`) inherits the upstream's path-resolution rejection of `..` and absolute paths. The first call in the algorithm to encounter the offending path fails before any side effect occurs. Pre-flight rejection — atomicity holds (per FR-010).
- **Path resolves to a folder**: `old_path` identifies a folder. The pre-flight `rest.getFileContents(old_path)` fails because folders aren't readable as file content; the upstream error propagates per FR-001a. Pre-flight rejection — atomicity holds.
- **Embed link integrity (attachments)**: `old_path` is a non-markdown attachment (e.g. `images/cover.png`) referenced by `![[cover.png]]` embed links elsewhere in the vault. Pass C of the wikilink-rewrite step (`!\[\[(<old-basename>)(\|[^\]]*)?\]\]`) handles embed forms with the same correctness as Pass A handles the bare-and-aliased forms.
- **Mid-flight failure (write_destination, find_and_replace_*, delete_source)**: Any failure after the source content has been read leaves the vault in a partial state. The wrapper does NOT attempt automated recovery. The structured response identifies which step failed and what was written, so the caller (or a human via `git status` / `git restore .`) can recover. The git-clean precondition (Acceptance Scenario US3.2) is the rollback baseline.
- **Filenames with regex metacharacters**: Names containing `(`, `)`, `.`, `+`, `*`, etc. (e.g. `Obsidian MCP Server (Multi-Vault Edition).md`) MUST be regex-escaped before substitution into the wikilink-rewrite patterns. The implementation includes an `escapeRegex(string)` utility that runs on the derived `<old-basename>` and `<old-folder>` strings before they are inserted into the regex templates (research.md §R10).
- **Same-folder rename**: `dirname(old_path) == dirname(new_path)`. Pass D (full-path-form rewrite) is skipped because no full-path references could have changed — only the basename did.
- **Unicode and special-character filenames**: Names containing spaces, non-ASCII characters, or characters legal in the vault but awkward at the OS level must be passed through unchanged so the underlying REST endpoints handle them.
- **Vault not on clean git state at invocation time**: The wrapper does NOT check git state. The git-clean precondition is a documented caller responsibility — `find_and_replace` (item 25) imposes the same precondition for the same reason, and `rename_file` inherits it.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The MCP server MUST expose a tool named `rename_file` that accepts two required string parameters, `old_path` and `new_path`, both interpreted as vault-relative paths to a file (markdown note or attachment such as image, PDF, audio).
- **FR-001a**: When `old_path` resolves to a folder rather than a file, `rename_file` MUST result in an error and MUST NOT modify any file or folder. The rejection manifests via the pre-flight `rest.getFileContents(old_path)` call failing for a folder path; the upstream error is propagated verbatim per FR-008. The tool MUST NOT perform a pre-flight folder-vs-file check of its own. (Folder-renaming is the responsibility of a future, separately-named `rename_folder` tool.)
- **FR-002**: `rename_file` MUST perform the rename via wrapper-side composition over the existing service-layer REST primitives — `rest.getFileContents`, `rest.putContent`, `rest.listFilesInDir`, `rest.findAndReplace` (provided by Tier 2 backlog item 25), and `rest.deleteFile` — invoked in the order specified in [contracts/rename_file.md §"Composition algorithm"](./contracts/rename_file.md). The tool MUST NOT dispatch any Obsidian command via `POST /commands/{commandId}/`; the 2026-05-02 T002 spike established that no stock Obsidian command performs a programmatic rename when dispatched headlessly. The tool MUST NOT issue filesystem-level rename system calls of its own.
- **FR-003**: `rename_file` MUST be implemented as a thin composition over the per-vault `ObsidianRestService` methods named in FR-002. It MUST NOT contain markdown-AST parsing, link-graph construction, or any other logic that depends on the markdown's parsed structure rather than its byte content. Wikilink rewrites MUST be performed by `rest.findAndReplace` (item 25), invoked with regex patterns the wrapper constructs but does not interpret.
- **FR-004**: When the rename succeeds, every `[[wikilink]]` and `![[embed]]` reference in the vault that resolved to the old basename (or, for cross-folder renames, the old full path) MUST be updated to reference the new path. Coverage is per the wikilink shape catalogue in FR-014; updates are performed via four regex passes through `rest.findAndReplace`. The Obsidian "Automatically update internal links" setting is **irrelevant** under this implementation — the integrity guarantee comes from the wrapper's own regex passes, not from Obsidian's index.
- **FR-004a**: When an updated wikilink originally carried a user-supplied alias (`[[target|alias]]` or `![[target|alias]]`), the alias MUST be preserved in the rewritten reference. The regex patterns in Passes A, B, and C capture the alias segment as `$2` (or `$3` in Pass B) and emit it verbatim in the replacement. Verification surface: User Story 1, Acceptance Scenario 2; unit tests for each pass.
- **FR-005**: The tool's MCP description (the `description` field exposed in the tool schema) MUST explicitly state: (a) that the operation is multi-step and not atomic; (b) that the caller SHOULD invoke against a clean git working tree, since `git restore .` is the rollback baseline; (c) which wikilink shapes are reliably rewritten (the FR-014 catalogue); and (d) that the Obsidian "Automatically update internal links" setting is **irrelevant** under this implementation (no need to enable it; no effect when toggled).
- **FR-006**: When `new_path` already identifies an existing file in the vault, `rename_file` MUST refuse to proceed, returning a wrapper-constructed error of the form `"destination already exists: <new_path>"`. The check is the pre-flight `rest.getFileContents(new_path)`: a 200 response triggers the rejection; a 404 response allows the algorithm to proceed. The wrapper MUST NOT issue any `putContent`, `findAndReplace`, or `deleteFile` call when the collision check rejects. (This is the single Q1 supersession — see Clarifications.)
- **FR-007**: When `old_path` does not identify an existing file in the vault, `rename_file` MUST return an error and MUST NOT create any file at `new_path`. The check is the pre-flight `rest.getFileContents(old_path)`: a 404 response is propagated verbatim per FR-008 (Q1 still applies — error message text comes from upstream, not the tool).
- **FR-008**: `rename_file` MUST surface failures from any composed REST primitive (`rest.getFileContents`, `rest.putContent`, `rest.listFilesInDir`, `rest.findAndReplace`, `rest.deleteFile`) to the caller, and MUST NOT report success unless every step in the composition algorithm completed successfully. The wrapper MUST NOT swallow individual-primitive failures into success text or fallback values (Constitution Principle IV; this is the property the original Option-A design was protecting via the service-layer composition rationale, and it carries forward unchanged into Option B).
- **FR-009**: When `old_path` and `new_path` are equal (after `.trim()`), `rename_file` MUST return a non-error result without issuing any REST call.
- **FR-010**: When either `old_path` or `new_path` resolves outside the focused vault, `rename_file` MUST result in an error and MUST NOT modify any file. Detection of out-of-vault paths is delegated to the underlying REST endpoints' path resolution; the first composed call to encounter the offending path fails and the resulting error is propagated verbatim per FR-008. The tool MUST NOT perform a pre-flight `..`/absolute-path filter of its own.
- **FR-011**: On success, the tool's response MUST be a structured object containing at minimum: `ok: true`, the original and resulting paths (`oldPath`, `newPath`), the `wikilinkPassesRun: string[]` array naming which passes executed (subset of `["A", "B", "C", "D"]`), and the `wikilinkRewriteCounts` object reporting how many references each pass rewrote. Per FR-014, this lets the caller confirm what changed without a follow-up read AND distinguish "no passes ran" from "passes ran, found zero references." On failure after step 4 (mid-flight), the response MUST include `ok: false`, a `failedAtStep` string identifying which step failed (one of: `pre_flight_source`, `pre_flight_destination`, `pre_flight_parent`, `read_source`, `write_destination`, `find_and_replace_pass_A` / `_B` / `_C` / `_D`, `delete_source`), and a `partialState` object naming what was written and what was not.
- **FR-012**: `rename_file` MUST NOT create missing parent folders for `new_path`. The pre-flight check is `rest.listFilesInDir(dirname(new_path))`: a 404 response is propagated verbatim per FR-008. The wrapper MUST NOT expose a parameter (e.g. `create_parents`) that opts into folder creation; callers needing a missing folder must create it through a separate, explicitly-named tool before calling `rename_file`.
- **FR-013**: `rename_file` requires `rest.findAndReplace` from the per-vault service-layer module, shipped by Tier 2 backlog item 25. The dependency is **build-time**, not runtime — the build fails (or `RENAME_FILE_TOOLS` is not registered into `ALL_TOOLS`) if `rest.findAndReplace` is unavailable. The wrapper does NOT runtime-check for the presence of `find_and_replace`; the import is a static module dependency, mirroring how the original Option-A design was going to import `rest.openFile` and `rest.executeCommand`. Implementation order: item 25 ships first; item 4 (this feature) ships against the now-importable `rest.findAndReplace` afterward.
- **FR-014**: The tool's MCP description (per FR-005(c)) and the implementation (Passes A–D) MUST cover the following wikilink shapes:
  - **Reliable** (always rewritten when the target file is renamed): `[[basename]]`, `[[basename|alias]]`, `[[basename#heading]]`, `[[basename#heading|alias]]`, `[[basename#^block-id]]`, `![[basename]]`, `![[basename|alias]]`.
  - **Reliable when cross-folder rename** (Pass D): full-path forms `[[old-folder/basename]]`, `[[old-folder/basename#heading]]`, `[[old-folder/basename|alias]]`, etc. Pass D is skipped on same-folder renames (no full-path references could have changed).
  - **Less reliable / out of scope for this implementation**: relative-path forms (`[[../folder/basename]]`) — the regex passes do not enumerate every parent-directory traversal pattern. Markdown-style links of the form `[text](path.md)` are also out of scope.
- **FR-015**: The composition is **multi-step and not atomic.** Failure at or after step 5 (`write_destination`) leaves the vault in a partial state. The wrapper MUST NOT attempt automated recovery (no reverse-direction find_and_replace, no delete-of-newly-written-destination). The structured response surface from FR-011 is sufficient for caller-side or human-driven recovery. The git-clean precondition (FR-005(b)) is the documented rollback baseline; `git restore .` from the pre-call commit fully reverses any partial state.

### Key Entities *(include if feature involves data)*

- **Vault file**: A note or attachment located at a vault-relative path. Identified by its path. Has zero or more incoming wikilinks from other vault files.
- **Wikilink reference**: An occurrence of `[[target]]`, `[[target|alias]]`, `[[target#heading]]`, `[[target#heading|alias]]`, `[[target#^block-id]]`, `![[target]]`, or `![[target|alias]]` inside a vault file's body, where `target` is a basename or a `folder/basename` path. Wikilinks live in markdown prose only — occurrences inside fenced code blocks or HTML comments MUST NOT be rewritten (`find_and_replace` (item 25) honors `skipCodeBlocks: true` and `skipHtmlComments: true` flags that the wrapper sets).
- **Composition step**: One of the eight named steps in [contracts/rename_file.md §"Composition algorithm"](./contracts/rename_file.md): `pre_flight_source`, `pre_flight_destination`, `pre_flight_parent`, `read_source`, `write_destination`, `find_and_replace_pass_A` / `_B` / `_C` / `_D`, `delete_source`. Each step is a single REST call (or no-op for skipped Pass D); the structured response (FR-011) names the step at which a mid-flight failure occurred.
- **Wikilink rewrite pass**: One of four regex passes (A, B, C, D) executed against the vault by `rest.findAndReplace`. Each pass targets a distinct wikilink shape family per FR-014. The wrapper builds the regex strings using the `<old-basename>` (and, for Pass D, `<old-folder>`) derived from `old_path`, both passed through `escapeRegex` to neutralise regex metacharacters.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: For each "reliable" wikilink shape in FR-014 (bare, aliased, heading-targeted, heading-targeted-aliased, block-reference, embed, embed-with-alias, plus full-path forms when cross-folder), 100% of vault-wide references to the renamed file's old basename (or old full path, for Pass D) are rewritten by a single `rename_file` call. Verified by hermetic regex-pass unit tests against synthetic strings AND by the manual quickstart Part 2 against a controlled fixture file.
- **SC-002**: An MCP-aware agent reading only the tool catalogue (no source code, no external docs) can correctly identify (a) the multi-step / non-atomic nature, (b) the git-clean precondition, (c) the wikilink shape coverage list, (d) the irrelevance of the Obsidian setting. The description must contain each of the four disclosures verbatim — pinned by registration tests.
- **SC-003**: For **pre-flight rejections** (FR-006 collision, FR-007 source-missing, FR-001a folder, FR-010 out-of-vault, FR-012 missing-parent), zero files in the vault are modified — the vault is byte-for-byte identical to its pre-call state. **For mid-flight failures** (FR-015 — failures at or after `write_destination`), the operation is explicitly best-effort: the vault may be in a partial state, the structured response identifies which step failed and what was written, and `git restore .` from the pre-call commit is the documented rollback. SC-003 does NOT promise atomicity for mid-flight failures.
- **SC-004**: A caller receives the full structured response (`ok`, `oldPath`, `newPath`, `wikilinkPassesRun`, `wikilinkRewriteCounts`, plus `failedAtStep` / `partialState` on mid-flight failure) in a single MCP tool-call result. The caller does not need to issue any follow-up read to know what changed.
- **SC-005**: The implementation MUST NOT parse markdown semantically — no AST traversal, no link-graph construction, no logic that depends on the markdown's parsed structure rather than its byte content. Reading file bytes via the existing REST primitives (`rest.getFileContents`, etc.) is allowed for the purpose of moving content from `old_path` to `new_path`. Constructing wikilink-shape regex patterns and passing them as opaque strings to `rest.findAndReplace` is allowed. The wrapper does not interpret what the regexes match — that's `find_and_replace`'s job, not `rename_file`'s. Enforced by the SC-005 import-guard test, narrowed per the Option-B redesign to forbid imports from any markdown-AST library, HTML/XML parser, or link-graph traversal library.

## Assumptions

- The Obsidian instance the MCP server is connected to has the Local REST API plugin (`coddingtonbear/obsidian-local-rest-api`) enabled and reachable.
- Tier 2 backlog item 25 (`find_and_replace`) ships first and exposes a service-layer helper `rest.findAndReplace(...)` that accepts a regex pattern, a replacement string, and at minimum the `skipCodeBlocks: true` and `skipHtmlComments: true` flags. The exact signature is item 25's contract; this feature consumes it.
- The user (not the tool) is responsible for ensuring the vault is on a clean git working tree before invoking `rename_file`. This is the same precondition `find_and_replace` (item 25) imposes for its own destructive operations, and `rename_file` inherits it. The tool documents the requirement in its description but does not programmatically check git status.
- Markdown-style links of the form `[text](path.md)` are out of scope — Passes A–D do not target them. Callers requiring update of such links must do so through a separate composition.
- Relative-path wikilinks of the form `[[../folder/basename]]` are out of scope — Passes A–D do not enumerate parent-directory traversal patterns. Callers needing this must perform additional `find_and_replace` calls themselves.
- "Vault-wide" means every file in the focused vault that `find_and_replace` (item 25) considers in scope. The wrapper inherits item 25's vault-traversal scope and does not extend or narrow it.
- Restoring Option A (Obsidian-managed rename via a programmatic `POST /vault/{path}/rename`-style endpoint with body parameters) is captured as backlog item 28, deferred pending an upstream `coddingtonbear/obsidian-local-rest-api` plugin enhancement. Out of this project's direct control.
